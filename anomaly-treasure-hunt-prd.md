# 이상현상 조사형 보물찾기 웹앱 기획서 / Codex 개발 지시서 v0.2

## 0. 문서 목적

이 문서는 “추리형 보물찾기 웹앱”을 단순 정답 맞히기 이벤트가 아니라, SCP 재단풍 / 백룸풍 / 도시괴담풍의 **이상현상 조사형 웹앱**으로 구현하기 위한 제품 기획서 겸 개발 지시서다.

Codex는 이 문서를 기준으로 MVP를 구현한다.

핵심 방향은 다음과 같다.

> 사용자는 문제를 푸는 사람이 아니라, 도시 곳곳에서 발생한 이상현상 기록을 열람하고, 발생 위치를 식별한 뒤, 현장 증거 사진을 보고하는 조사원이다.

---

## 1. 서비스 개요

### 서비스 콘셉트

도시 곳곳에는 평범해 보이지만 특정 조건에서 이상 징후를 드러내는 장소가 있다.

사용자는 매주 공개되는 “사건 기록”을 열람한다.

사건 기록에는 다음 정보가 포함된다.

- 이상현상 개요
- 최초 보고 내용
- 시스템 오류 로그
- 관측 기록
- 위치 추론 단서
- 조사 지시
- 촬영 조건
- 식별 코드

사용자는 기록을 해석해 실제 장소를 찾아내고, 현장에서 지정 조건에 맞는 사진을 촬영해 제출한다.

정상적으로 보고가 완료되면 보상 추첨 대상에 등록된다.

## 2. 핵심 차별점

일반적인 경품 이벤트는 다음과 같은 방식이다.

- 댓글 달기
- 팔로우하기
- 친구 태그하기
- 출석 체크하기
- 룰렛 돌리기

이 서비스는 사용자의 행동을 다음처럼 바꾼다.

| 일반 이벤트 | 이 서비스 |
|---|---|
| 이벤트 참여 | 사건 기록 열람 |
| 문제 풀기 | 이상현상 조사 |
| 정답 입력 | 위치 식별 |
| 인증 | 현장 증거 보고 |
| 당첨자 | 보상 대상자 |
| 지난 이벤트 | 보관 기록 |

핵심은 “치킨 기프티콘 이벤트”가 아니라, **보상이 붙어 있는 도시 미스터리 조사 콘텐츠**다.

## 3. 세계관

### 세계관 한 줄 정의

도시의 일부 장소는 특정 조건에서 비정상적인 기록 오류를 발생시킨다.  
사용자는 공개 조사 파일을 열람하고, 현장에서 증거를 확보하는 조사원 역할을 수행한다.

### 서비스 내 설정

이 웹앱은 표면적으로는 이벤트 서비스지만, 내부 세계관상으로는 **이상현상 기록 보관소**다.

사용자는 매주 하나의 신규 파일을 열람한다.

각 파일은 특정 장소에서 발생한 비정상 현상을 다룬다.

예시:

- 특정 공원에서 사진 파일명이 자동으로 바뀜
- 특정 통로에서 지도 경로가 반복됨
- 특정 구조물에서 비어 있는 공간의 음향이 감지됨
- 특정 카페거리에서 동일한 간판이 여러 번 기록됨
- 특정 정원에서 사진의 촬영 시간이 00:00:00으로 고정됨

이상현상은 실제 위험이 아니라 허구의 분위기 장치다.

## 4. 컨셉 방향

### 참고 감성

이 서비스는 다음 감성을 참고한다.

- SCP 재단풍 격리 문서
- 백룸풍 리미널 스페이스
- 도시괴담

직접 차용이 아니라, 자체 세계관으로 재해석한다.

### 금지 방향

- 자살, 납치, 살인, 실종을 현실적으로 묘사
- 고어
- 과도한 공포 연출
- 위험 장소 방문 유도
- 사유지 침입 유도

## 7. 기본 플로우

### 사용자 플로우

1. 사용자가 웹앱에 접속한다.
2. 홈에서 이번 주 신규 사건 기록을 확인한다.
3. 사건 상세 페이지에서 이상현상 보고서를 읽는다.
4. 단서와 조사 지시를 바탕으로 실제 장소를 추론한다.
5. 현장에 방문한다.
6. 지정 조건에 맞는 사진을 촬영한다.
7. 사진과 식별 코드를 제출한다.
8. 서버가 제출 가능 여부와 코드 일치 여부를 검증한다.
9. 보고가 완료되면 추첨 대상자로 등록된다.
10. 이벤트 종료 후 보상 대상자가 발표된다.

### MVP에서는 선택 가능한 방식

MVP에서는 운영 부담을 줄이기 위해 두 가지 제출 방식을 둘 수 있다.

#### 방식 A. 코드 제출 중심

- 사용자가 현장에서 확인한 코드를 입력한다.
- 사진 제출은 선택 사항 또는 v2 기능으로 둔다.

#### 방식 B. 사진 보고 중심

- 사용자가 현장에서 촬영한 사진을 업로드한다.
- 지정된 식별 코드를 함께 입력한다.
- 관리자가 사진을 사후 검토할 수 있다.

이 서비스의 컨셉에는 방식 B가 더 잘 맞는다.

MVP에서는 다음 방식을 추천한다.

```txt
사진 업로드 + 식별 코드 입력
```

단, 사진 내용의 자동 판별은 MVP에서 제외하고, 관리자가 필요 시 검토하는 방식으로 구현한다.

## 10. 사건 유형

### 10.1 사진 오류형

특정 장소에서 촬영된 사진의 파일명, 색상, 메타데이터, 분석 결과가 비정상적으로 기록되는 유형.

예시:

- 사진 파일명이 자동으로 바뀜
- EXIF 시간이 00:00:00으로 고정됨
- 특정 색상 채널이 과도하게 높게 기록됨
- 이미지 분류 결과가 이상하게 나옴

### 10.2 지도 오류형

지도, 길찾기, 좌표, 이동 경로가 비정상적으로 기록되는 유형.

예시:

- 길찾기가 같은 골목으로 반복됨
- 존재하지 않는 출구가 표시됨
- 좌표가 근처 다른 장소로 튐
- 도보 경로가 폐곡선을 그림

### 10.3 음향 잔류형

특정 구조물이나 공간에서 비정상적인 음향 로그가 감지되는 유형.

예시:

- 빈 공간에서 저주파 감지
- 발소리가 늦게 따라옴
- 녹음에는 없는데 분석 로그에는 소리가 있음
- 반복적인 기계음이 기록됨

### 10.4 경계공간형

백룸풍의 리미널 스페이스 감성을 활용하는 유형.

예시:

- 끝나지 않는 복도
- 비슷한 문이 반복되는 건물
- 사람이 많은데 이상하게 조용한 장소
- 출입구가 많지만 방향감이 흐려지는 공간

### 10.5 사물단서형

특정 구조물, 간판, 조형물, 색상, 표식을 중심으로 위치를 추적하는 유형.

예시:

- 붉은 벽돌
- 오래된 굴뚝
- 원형 탱크
- 유리 온실
- 파란 문
- 둥근 시계탑
- 특정 안내판

## 11. 문제 난이도 설계

MVP에서는 너무 어렵게 만들지 않는다.

### 권장 난이도

- 1차 단서: 지역 범위 암시
- 2차 단서: 장소의 과거 용도 또는 분위기 암시
- 3차 단서: 특정 구조물 또는 표식 암시
- 4차 단서: 현장에서 찍어야 할 대상 제시

### 목표 정답률

| 난이도 | 목표 정답률 |
|---|---|
| 쉬움 | 60~80% |
| 보통 | 30~50% |
| 어려움 | 10~20% |

초기 MVP는 “보통보다 약간 쉬움”을 목표로 한다.

이유:

- 초반 이탈을 줄여야 함
- 콘텐츠 구조를 학습시켜야 함
- 너무 어려우면 방문/제출까지 이어지지 않음

## 13. 이벤트 운영 구조

### 이벤트 주기

매주 1회 공개한다.

예시:

- 토요일 12:00 신규 사건 공개
- 토요일 12:00 ~ 일요일 23:59 보고 가능
- 월요일 12:00 보상 대상자 발표
- 월요일 중 기프티콘 발송

### 참여 방식

1. 신규 사건 공개
2. 사용자가 기록 열람
3. 발생 위치 추론
4. 현장 방문
5. 지정 조건에 맞는 사진 촬영
6. 식별 코드 입력
7. 보고 완료
8. 정답자 중 보상 대상자 추첨

### 당첨 방식

- 보고 완료자 중 1명 랜덤 추첨
- 동일 계정은 이벤트당 1회만 보고 완료 가능
- 부정 제출 의심 계정 제외 가능
- 관리자가 사진 보고를 반려할 수 있음

선착순 방식은 제외한다.

이유:

- 사용자가 급하게 이동할 수 있음
- 사고 위험 증가
- 봇/대리참여 문제가 생김
- 분쟁 대응 부담이 큼

- 낮 시간대 방문이 안전한 장소

금지:

- 출입금지 구역
- 사유지
- 주거지
- 공사장
- 폐건물
- 철도/도로 위험 구역
- 야간 방문이 필요한 장소
- 관리자의 허가 없이 QR을 붙여야 하는 장소

### 촬영 정책

- 타인의 얼굴이 크게 나오지 않도록 안내한다.
- 차량 번호판, 개인 정보, 민감한 시설이 노출되지 않게 안내한다.
- 위험한 각도, 난간, 도로 한복판 촬영을 요구하지 않는다.
- 실내 시설의 경우 촬영 가능 여부를 사전에 확인한다.
- MVP에서는 공공장소의 안내판, 표식, 외부 구조물 중심으로 설계한다.

현재 진행 중인 사건 조회.

#### GET `/api/cases`

지난 사건 목록 조회.

#### GET `/api/cases/:id`

사건 상세 조회.

포함 데이터:

- 사건 기본 정보
- 사건 본문
- 단서 목록
- 조사 지시
- 촬영 조건
- 안전 안내
- 제출 가능 여부

#### POST `/api/cases/:id/report`

사진 보고 및 식별 코드 제출.

요청:

```json
{
  "code": "GREEN-LUNG-001",
  "photoPath": "storage/path/example.jpg"
}
```

처리 조건:

- 로그인 필수
- 사건 진행 기간 내 제출
- 제출 횟수 제한
- 코드 정규화 후 해시 비교
- 사진 업로드 여부 확인
- 이미 승인된 보고가 있으면 재제출 불가
- Cloudflare Turnstile 검증 가능 구조 마련

반환 예시:

```json
{
  "isCodeCorrect": true,
  "reviewStatus": "pending",
  "message": "보고가 접수되었습니다. 검토 후 추첨 대상 등록 여부가 확정됩니다."
}
```

#### GET `/api/cases/:id/result`

보상 대상자 발표 조회.

#### GET `/api/me/reports`

내 보고 이력 조회.

## 24. 샘플 사건 기록

## FILE-001: 도시폐 호흡 오류

### 열람 등급

공개 관측 허가

### 상태

조사 가능

### 보상

치킨 기프티콘 추첨 대상 등록

---

### 사건 개요

서울 동부 권역에서 반복적인 이미지 분류 오류가 보고되었다.

제보자들은 모두 평범한 녹지 사진을 제출했다.

그러나 내부 분석 시스템은 해당 사진들을 전부 같은 이름으로 저장했다.

```txt
LUNG_AREA_001.jpg
LUNG_AREA_001.jpg
LUNG_AREA_001.jpg
```

사진 속에는 특별한 물체가 없었다.

나무, 산책로, 낮은 건물, 벤치, 지나가는 사람들.

그러나 분석 로그는 해당 장소를 계속해서 다음과 같이 분류했다.

```txt
도시형 호흡기관
비정상 녹지 밀도
회색 구조물 내부의 폐포 반응
```

---

### 최초 보고

최초 오류는 토요일 13:12에 발생했다.

한 사용자가 성수동 인근에서 촬영한 사진을 업로드했으나, 파일명은 자동으로 변경되었다.

```txt
IMG_4821.jpg → LUNG_AREA_001.jpg
```

이후 같은 구역에서 촬영된 사진 17장이 동일한 방식으로 변환되었다.

공통점은 하나였다.

사진 속 어딘가에 반드시 “숲”이라는 글자가 포함되어 있었다.

---

### 이상 징후

### 사건 개요

서울 서북권의 한 공원에서 비정상적인 음향 잔류 현상이 보고되었다.

제보자들은 모두 같은 설명을 남겼다.

```txt
안에는 아무것도 없는데,

```

해당 장소는 과거 액체 연료를 보관하던 시설이었다.

현재는 시민에게 개방된 문화 공간으로 전환되었으나, 내부 음향 분석 결과 일부 구조물에서 반복적인 저주파가 감지되었다.

시스템은 해당 구조물을 다음과 같이 분류했다.

```txt
EMPTY_TANK_ACTIVE
```

### 관측 기록

관측 대상은 원형의 대형 구조물이다.

외부에서는 공원처럼 보이나, 내부 구조는 일반적인 건축물과 다르다.

벽면은 둥글고, 소리는 짧게 사라지지 않는다.

일부 방문자는 같은 지점을 지나며 다음과 같은 착각을 경험했다.

- 방금 지나온 입구가 다른 방향에 있는 것처럼 느껴짐
- 발소리가 한 박자 늦게 따라오는 느낌
- 빈 공간 안쪽에 무언가 남아 있는 듯한 압박감
- 사진을 찍으면 원형 구조물이 화면 중앙으로 끌려 들어오는 느낌

### 사건 개요

한강 인근의 특정 공원에서 물의 흐름이 기록되지 않는 오류가 발생했다.

지도상으로는 공원이다.

그러나 내부 구조 분석 결과, 해당 장소는 공원보다 시설에 가깝다.

과거에는 물을 정화하던 곳이었다.

현재는 사람들이 산책하고 사진을 찍는 곳이다.

문제는 이곳에서 촬영된 일부 사진의 시간 정보가 멈춘다는 점이다.

```txt
00:00:00
```

시스템은 해당 장소를 다음과 같이 분류했다.

```txt
WATER_MEMORY_GARDEN
```

- 물이 흐르지 않는 것처럼 보이는 구조물이 반복 촬영됨
- 콘크리트 구조물과 정원 요소가 함께 기록됨
- 일부 사진에서 촬영 시간이 00:00:00으로 고정됨
- “정수”와 “정지”가 동일 키워드로 분류됨

- 물, 콘크리트 구조물, 정원 요소 중 하나 이상 포함
- 타인의 얼굴이 크게 나오지 않도록 촬영

## 27. 보안 로직

### 식별 코드 정규화

```ts
export function normalizeCode(code: string) {
  return code.trim().toUpperCase().replace(/\s+/g, "");
}
```

### 해시 처리

- 식별 코드는 서버에서만 해시 처리한다.
- DB에는 해시만 저장한다.
- 클라이언트에는 정답 코드 해시나 정답 위치를 내려주지 않는다.

### 관리자 권한 체크

- `profiles.role`이 `admin`인 사용자만 관리자 API 접근 가능
- 서버 API에서 반드시 재검증
- 클라이언트 라우팅만으로 보호하지 않는다

### RLS 정책

필수 정책:

- 사용자는 자신의 보고만 조회 가능
- 사용자는 자신의 보고만 생성 가능
- 사용자는 사건 기록 중 published 상태만 조회 가능
- 관리자는 모든 사건, 보고, 보상 정보를 조회/수정 가능
- 제출 사진은 관리자와 제출자만 접근 가능

## 34. 최종 정체성

이 서비스는 “치킨을 주는 이벤트”가 아니다.

정확한 정체성은 다음과 같다.

> 치킨 기프티콘을 보상으로 제공하는 도시 이상현상 조사 콘텐츠.

사용자는 정답자가 아니라 조사원이다.

장소는 정답이 아니라 발생 위치다.

사진은 인증샷이 아니라 증거 보고다.

이 방향을 유지해야 서비스 컨셉이 살아난다.
