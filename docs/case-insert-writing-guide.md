# Case Insert Writing Guide

이 문서는 이상현상 보물찾기 `cases` 레코드 초안을 작성할 때 참고하는 컨셉 가이드다. 목적은 실제 장소 기반 미션을 유지하면서도, 사건 파일 텍스트를 SCP 관측 보고서, 백룸/리미널 스페이스, 나폴리탄 괴담의 결로 일관되게 작성하는 것이다.

## 전체 컨셉

- 현실 장소를 기반으로 쓴다.
- 시작은 평범한 안내문처럼 보이게 한다.
- 읽을수록 규칙과 예외가 이상해지는 구조를 유지한다.
- 과도한 설정 설명보다 관측 규칙, 승인 조건, 금지 조건을 앞세운다.
- 참가자가 실제로 안전하게 수행 가능한 현장 미션이어야 한다.

핵심 문장 감각:

- 기록
- 관측
- 분류
- 승인
- 폐기
- 반환
- 표식
- 반복
- 빈 공간
- 내부 공간
- 출구
- 산책로
- 낮은 건물
- 안내판
- 같은 벤치
- 너무 긴 길
- 비어 있는 사진
- 제출하지 않은 사람
- 일반 풍경으로 반환됨

## 문체 원칙

기본 톤:

- 감정적으로 쓰지 않는다.
- 단정적이고 사무적인 문장으로 쓴다.
- 설명보다 판정과 절차를 우선한다.

권장 문장 구조:

1. 정상적인 안내
2. 이상한 예외
3. 단호한 금지 또는 폐기 조건

예시:

```text
해당 구역은 일반적인 공원입니다.
산책로, 벤치, 수목, 방문객이 존재합니다.
다만 사진으로 저장된 이후에는 일부 이미지가 실외가 아닌 내부 공간으로 분류됩니다.

관측자는 반드시 나무를 촬영해야 합니다.
나무가 없는 사진은 공원으로 분류되지 않습니다.
공원이 아닌 사진은 승인되지 않습니다.
```

## 피해야 할 표현

- 정답을 직접 말하는 게임식 표현
- 단서를 친절하게 해설하는 문장
- 괴물, 귀신, 저주, 마법, 차원문 같은 과한 판타지 표현
- 사유지, 공사장, 야간 통로, 출입금지 구역 접근을 유도하는 문장

대신 다음 표현을 우선한다:

- 분류 오류
- 내부 공간
- 반복 구간
- 누락된 기록
- 승인 불가

## 필드별 작성 기준

### `summary`

- 짧게 사건의 이상성을 설명한다.
- 구조는 `현실 장소 + 평범한 사진 + 저장 이후 이상 분류 + 분석기가 요구하는 조건`을 따른다.

### `report_body`

- 가장 중요한 분위기 필드다.
- 번호 규칙 형식으로 작성한다.
- 시작 문단에서 공개 관측 지침임을 명시한다.
- 중간에는 정상 규칙과 이상 예외를 섞는다.
- 마지막에는 코드와 승인 조건을 명확히 닫는다.

### `safety_notice`

- 현실 안전 기준을 모호하지 않게 쓴다.
- 공개 동선, 타인 얼굴 비식별, 사유지/공사장/야간 저시인 환경 금지를 분명히 적는다.

### `answer_location`

- 실제 정답 장소를 기록한다.
- 이 값은 운영/서버 측 기준값이다. 클라이언트 노출 경로에 실리지 않게 유지한다.

### `identification_code`

- 참가자가 입력할 짧은 코드 초안을 정한다.
- 장소나 촬영 대상과 연결된 짧은 영어 단어가 적합하다.
- 구현 시에는 백엔드 규칙에 맞게 서버에서 해시 저장한다. 문서 예시는 사람이 검토하기 위한 평문 초안으로 다룬다.

### `completion_message`

- 승인 성공 사실을 분명하게 알려야 한다.
- 분위기는 유지하되 결과는 명확해야 한다.

### `clues_json`

- 정답을 직접 공개하지 않고 장소와 촬영 대상을 유도해야 한다.
- 권장 순서는 `관측 구역`, `주변 잔류 기록`, `촬영 대상`, `이름 조각`이다.

### `mission_instruction`

- 참가자가 실제로 해야 할 행동을 분명하게 쓴다.
- 촬영 대상, 현장 맥락, 승인 불가 예시, 코드 입력을 한 문단 안에서 정리한다.

### `mission_photo_requirement`

- 사진 검증 기준을 가장 명확하게 쓴다.
- 무엇이 반드시 보여야 하는지와 무엇이 반려되는지를 구체적으로 적는다.

### `mission_caution`

- 실제 안전 지침을 우선한다.
- 괴담 톤은 약하게만 섞고, 행동 기준은 분명하게 유지한다.

## 최종 체크리스트

기능 체크:

- 실제 정답 장소가 `answer_location`에 들어갔는가
- 참가자 입력 코드 초안이 `identification_code`에 들어갔는가
- 사진으로 찍어야 할 대상이 명확한가
- 장소명 표식이 필수인지 아닌지 일관적인가
- 미션 조건과 단서가 서로 모순되지 않는가
- 안전 안내가 실제 행동 기준으로 충분히 명확한가

분위기 체크:

- SCP식 관측 보고서 느낌이 있는가
- 백룸/리미널 스페이스식 빈 공간, 반복, 내부 공간 감각이 있는가
- 나폴리탄 괴담처럼 규칙문 형식을 유지하는가
- 설명보다 금지사항과 예외 조건이 중심인가
- 정답을 너무 직접적으로 말하지 않는가
- 과하게 판타지스럽지 않은가

## 작성 공식

1. 현실 장소 하나를 정한다.
2. 그 장소의 평범한 요소 하나를 촬영 대상으로 정한다.
3. 분석기가 그 요소를 비정상적으로 분류한다고 설정한다.
4. 참가자에게는 현실적으로 안전한 촬영 조건을 준다.
5. 본문은 번호가 붙은 안전수칙처럼 쓴다.
6. 중간중간 이상한 예외를 넣는다.
7. 마지막에는 코드와 승인 조건을 명확히 쓴다.

## 구현 메모

- 작성 단계에서는 `answer_location`, 평문 `identification_code` 초안, `clues_json`을 함께 검토해 단서와 정답이 모순되지 않는지 확인한다.
- 서버 저장 단계에서는 백엔드 규칙에 맞게 식별 코드를 정규화하고 해시 처리한다.
- 클라이언트 응답에는 정답 장소와 평문 코드가 노출되지 않게 유지한다.
- 시드 데이터나 운영 입력값을 만들 때도 실제 위험 행동을 유도하는 문장은 제외한다.

## 에이전트 응답 규칙

사용자가 다음처럼 요청하는 경우:

- `문제 하나 만들어줘`
- `케이스 insert 하나 만들어줘`
- `새 사건 파일 하나 작성해줘`

에이전트는 기본적으로 기존 시드 파일이나 앱 코드를 수정하지 않는다. 대신 바로 실행 가능한 독립형 SQL `INSERT` 1개만 반환한다.

기본 응답 계약:

1. 기존 `apps/api/src/modules/shared/infrastructure/sqlite-seed.ts`는 수정하지 않는다.
2. 기존 케이스 레코드를 덮어쓰지 않도록 새 `id`, 새 `file_no`, 새 `episode_no`를 사용한다.
3. 결과는 설명 위주 문서가 아니라, 복사 후 바로 실행 가능한 SQL 블록을 우선 제공한다.
4. 필요하면 SQL 아래에 아주 짧은 검토 메모만 덧붙인다.
5. 사용자가 별도로 요청하지 않으면 `UPDATE`, `UPSERT`, 마이그레이션 수정, 시드 수정은 하지 않는다.

  starts_at,
